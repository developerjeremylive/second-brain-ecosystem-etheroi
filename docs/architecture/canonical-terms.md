# Canonical Vocabulary

This document is the authoritative reference for all terminology used across the Second Brain Ecosystem. Every README, guide, architecture doc, PRD, and future website must use these terms consistently.

Last updated: 2026-05-17

## Core Concepts

| Term | Definition | Location |
|------|-----------|----------|
| **Second Brain** | A personal knowledge management system for capturing, organizing, and retrieving information. Based on Tiago Forte's BASB methodology. | Ecosystem-wide |
| **PARA** | Organizing system: Projects, Areas, Resources, Archive. Manages the user's life and projects. Librarian does not process PARA directly. | Vault root |
| **LLM Wiki pattern** | A persistent, growing wiki built incrementally by AI from raw sources. Articulated by Andrej Karpathy. Foundation of Librarian. | Guide 07, architecture |

## Vault Folders

| Term | Definition | Who writes |
|------|-----------|------------|
| **`inbox/`** | Temporary human capture. Unsorted notes to process during weekly reviews. Librarian never reads it directly. | You |
| **`daily/`** | Daily human notes. Not processed by Librarian by default. | You |
| **`1-projects/`, `2-areas/`, `3-resources/`, `4-archive/`** | PARA organization for the human Second Brain. Not processed by Librarian directly. | You |
| **`raw/`** | Immutable sources approved for Librarian to read. Moving or copying a file here marks explicit consent for AI processing. Not the same as `inbox/`, `daily/`, or PARA. | You (explicit consent) |
| **`wiki/`** | Structured knowledge curated by Librarian. Concept pages, entity pages, summaries, cross-references. The AI writes it; you read it. | Librarian |
| **`reviews/`** | Human-readable review and export surface. Librarian writes proposals here for you to review. | Librarian (you approve via CLI) |
| **`reports/`** | Vault diagnostics and derived artifacts generated by Librarian. | Librarian |
| **`memory/`** | Persistent agent/session continuity across sessions. | Librarian |
| **`configs/`** | Explicit, visible, editable configuration rules for Librarian. | You |
| **`.librarian/`** | Internal technical state: indexes, proposals, cache, locks. The proposal source of truth. | Librarian |
| **`_assets/`** | Attachments, images, and binary files used by notes. Not a Librarian source folder. | You / Obsidian |

## Librarian Operations

| Term | Definition |
|------|-----------|
| **proposal** | A pending change to the wiki. Generated by Librarian, stored in `.librarian/proposals/`. Must be reviewed and approved before applying. |
| **apply** | An authorized mutation. Executes a previously approved proposal, writing changes to the wiki. Only triggered via CLI (`librarian apply <id>`). |
| **approve** | User confirmation of a proposal via CLI (`librarian approve <id>`). Required before any wiki mutation. |
| **ingest** | Processing a new source from `raw/` into the wiki. Reads, extracts key ideas, creates/updates wiki pages, updates index and log. |
| **query** | Searching the wiki and synthesizing an answer with citations. Good answers may become new wiki pages. |
| **lint** | Periodic wiki health check: contradictions, stale claims, orphans, missing cross-references. |
| **index** | `wiki/index.md` — a catalog of all wiki pages with links, one-line summaries, and metadata. Updated on every ingest. |
| **log** | `wiki/log.md` — an append-only chronological record of ingests, queries, and lint passes. |
| **processed ledger** | Internal tracking of which sources have been ingested. Stored in `.librarian/processed.json`. Keyed by source path. Separate bounded context from proposal lifecycle. Written only after successful apply. |
| **operationId** | Unique correlation ID (`op_<ulid>`) generated per mutation attempt. Propagates through proposals, transactions, ledger, logs, and TUI. Each retry = new operationId. |
| **transaction** | A record of a single apply attempt in `.librarian/transactions/<operationId>.json`. Tracks targets, status, errors, and rollback state. |
| **rebuild** | Full index reconstruction via `librarian index rebuild`. Re-indexes all files, rebuilds all caches. |
| **retry** | Re-attempting a failed or rolled-back proposal via `librarian proposal retry <id>`. Generates new operationId. |
| **reset** | Manually returning a proposal to `pending` state via `librarian proposal reset <id>`. Not allowed for `applied` proposals. |

## Proposal States

| State | Meaning | Terminal |
|-------|---------|----------|
| **pending** | Created, awaiting approval | No |
| **applying** | Apply in progress | No |
| **applied** | Successfully applied | Yes |
| **failed** | Apply failed, may have partial writes | No |
| **rolled_back** | Apply failed + rollback completed | No |

## Index States

| State | Meaning |
|-------|---------|
| **fresh** | Index reflects current vault state |
| **stale** | Changes detected since last build |
| **missing** | No index exists |
| **rebuilding** | Rebuild in progress |

## System Description

**Canonical description of Librarian:**

> Librarian is a review-driven knowledge maintenance pipeline for Obsidian vaults. It is currently a tool-driven CLI in experimental alpha stage.

### Terms to use

- review-driven
- proposal-based
- manually triggered
- knowledge maintenance pipeline
- experimental alpha

### Terms to avoid until implemented

Do not use these terms in current-tense documentation until the corresponding features exist:

- ~~autonomous~~ (requires watchers, autonomous loops, recovery)
- ~~self-maintaining~~ (requires invalidation, reconciliation)
- ~~self-healing~~ (requires recovery, reconciliation engine)
- ~~automatic~~ (in context of maintenance; currently all operations are manually triggered)

## Key Distinctions

### `inbox/` vs `raw/`

- `inbox/` is temporary human capture. Fast dumping ground for unsorted notes.
- `raw/` is curated material explicitly approved for Librarian processing.
- Moving or copying a file from `inbox/`, `daily/`, or PARA into `raw/` is a **consent boundary**.
- Librarian never processes `inbox/`, `daily/`, or PARA directly.

### `reviews/` vs `.librarian/proposals/`

- `reviews/` is a human-readable review and export surface.
- `.librarian/proposals/` is the internal proposal source of truth.
- These are not the same thing. `reviews/` is what you read; `.librarian/proposals/` is what the system tracks.

### PARA vs Librarian layer

- PARA organizes your life and projects.
- Librarian organizes the AI-processable knowledge layer.
- They do not compete. They cooperate. PARA stays intact at the vault root; Librarian reads only sources explicitly placed in `raw/`.

### Processed ledger vs Proposal lifecycle

- Processed ledger answers: "Did we already ingest this source?" Keyed by source path.
- Proposal lifecycle answers: "What happened operationally with this proposal?" Keyed by proposal id.
- These are separate bounded contexts. The only coupling point is the apply pipeline.
- The ledger never knows about proposals. The proposal store never knows about the ledger.

## Privacy

**Canonical privacy wording:**

> When using local providers like Ollama, notes remain local. Cloud providers may receive relevant note fragments depending on configuration.

Do not use:

- ~~"your notes never leave your machine"~~ (inaccurate when cloud providers are configured)
- ~~"fully private"~~ (depends on configuration)
- ~~"zero data leaves your device"~~ (inaccurate for cloud setups)
